# Tali Core — Identity & Profile + Billing & Entitlements

Первый реальный кусок целевой архитектуры (Этап 7 аудита). Strangler-миграция:
этот сервис ничего не забирает у сегодняшнего бота автоматически — он просто
существует рядом, пока мы не переключим на него конкретные ноды в `ТАЛИ main`
одну за другой.

## Что внутри

- `POST /identity/resolve` — get-or-create внутреннего `user_id` по внешней
  идентичности (`telegram`/`email`/`apple`). Решает то, ради чего вообще
  затевался этот домен: `telegram_id` перестаёт быть первичным ключом человека.
- `GET /entitlements/:userId` — план, статус, лимит вопросов, дата окончания подписки.
- `POST /entitlements/:userId/increment-usage` — атомарный инкремент `questions_used`
  на уровне БД. Закрывает находку №10 (race condition при параллельной записи).
- `POST /webhooks/wayforpay` — приём колбэка оплаты с реальной проверкой подписи.
  Закрывает находки №0 (отсутствие проверки подписи) и №3 (секрет в открытом виде)
  одним движением — секрет живёт в переменной окружения, ничего не пишется без
  валидной подписи.

## Важно: HMAC WayForPay ещё не проверен на реальном колбэке

`src/wayforpaySignature.js` реализует алгоритм строго по документации WayForPay
(https://wiki.wayforpay.com/en/view/852102): HMAC-MD5, UTF-8, поля через `;`.

Честно: их же собственный пример в доках (для другого типа подписи — запроса
Purchase, не колбэка) не воспроизвёлся у меня побайтово ни в Node, ни в Python
(проверял независимо в обеих — они совпали друг с другом и с эталонным вектором
RFC 2104, значит сам примитив HMAC-MD5 работает верно; скорее всего у них в доках
просто устаревший/битый пример). Это не доказывает, что порядок полей для
**колбэка** (тот, что реально используется в `wayforpay.js`) на 100% совпадает
с тем, что шлёт их прод.

**Перед тем как подключать сюда реальные деньги:** обязательно прогони через
`POST /webhooks/wayforpay` настоящий тестовый колбэк из песочницы WayForPay
(тестовый мерчант, тестовая карта) и убедись, что `verifyCallbackSignature`
его принимает. Это не формальность — это единственный способ закрыть находку
№0 по-настоящему, а не на бумаге.

`npm run verify-hmac` — быстрый sanity-check, что HMAC-MD5 примитив в принципе
работает правильно (сверяется с эталонным вектором RFC 2104). Не заменяет
проверку на реальном колбэке выше.

## Локальный запуск

```bash
npm install
cp .env.example .env   # заполнить DATABASE_URL и WAYFORPAY_SECRET_KEY
npm run migrate         # применяет schema.sql к DATABASE_URL
npm start
curl localhost:3000/health
```

## Деплой на Render

Bodygraph-микросервис уже на Render — держим всё в одном месте.

1. Залить эту папку в git-репозиторий (GitHub/GitLab).
2. В Render: **New → Blueprint**, указать репозиторий — `render.yaml` сам
   создаст Postgres (`tali-core-db`) и веб-сервис (`tali-core-identity-billing`).
3. В настройках сервиса задать `WAYFORPAY_SECRET_KEY` вручную (он помечен
   `sync: false` в `render.yaml` — Render не хранит секреты в самом файле).
4. После первого деплоя выполнить `schema.sql` на новой базе — проще всего
   через `psql "$DATABASE_URL" -f schema.sql` с локальной машины, используя
   внешний connection string из Render.
5. Проверить `https://<service>.onrender.com/health`.

## Путь переключения — без остановки бота

Строго по strangler-плану из аудита: новый код не заменяет старое одним
переключателем, а забирает по одной ответственности.

1. **Сначала — ничего не трогая.** Сервис задеплоен, схема накатана, но ни
   один нод в n8n на него ещё не смотрит. Гоняем сюда синтетические запросы
   и настоящий тестовый WayForPay-колбэк (см. выше), чтобы набрать уверенность
   без риска для живых пользователей.
2. **WayForPay serviceUrl — первым.** Это самый повреждённый и самый
   независимый от остального узел. В личном кабинете WayForPay меняем
   `serviceUrl` на адрес нового `/webhooks/wayforpay`. Старый нод `Way for pay`
   в n8n не удаляем — выключаем, но оставляем как способ откатиться за минуту,
   если что-то пойдёт не так.
3. **Identity resolve — вторым.** В блоке онбординга (Блок 3, Этап 2) заменяем
   Airtable-поиск пользователя по `telegram_id` на вызов `/identity/resolve`.
   Первое время можно писать в оба места параллельно (Airtable остаётся
   источником правды, новый сервис — тенью), сверяя расхождения, прежде чем
   доверить ему запись единолично.
4. **Entitlements/increment-usage — третьим**, когда предыдущие два шага
   отработали без сюрпризов хотя бы неделю.
5. **Chart Service и Conversation/Knowledge** — по графику Этапа 7 (Chart —
   в любой момент независимо, Conversation/Knowledge — ближе к порогу 1000
   пользователей), в этот scaffold пока не входят.

## Отдельно и не входит в этот scaffold

Находки №1–2 (захардкоженный токен Telegram, захардкоженный секрет WayForPay
в самих нодах n8n) — это правки **внутри n8n** (перенести в n8n credentials),
не требуют миграции и не зависят от готовности этого сервиса. Стоит закрыть
их в любой момент, независимо от прогресса по остальному плану — если нужно,
могу разложить их отдельным быстрым чек-листом.
